Lesson 2: Claude Code - Flux de travail agentique et outils
Dans cette première leçon, nous allons passer en revue le flux de travail agentique (*agentic workflow*) de **Claude Code**, les outils qu'il utilise pour naviguer au sein de votre base de code, ainsi que la mémoire qu'il conserve d'une session à l'autre. Plongeons dans le sujet.

Voyons d'abord ce qu'est concrètement Claude Code. Lorsque nous parlons de systèmes agentiques, nous pensons généralement à un modèle, à un ensemble d'outils et à un environnement pour exécuter ces outils. Les modèles de langage excellent pour traiter des entrées et retourner des sorties. Cependant, dans bien des situations, ces modèles ne connaissent pas votre base de code, ne savent pas comment localiser des fichiers ni comment gérer des tâches multiples.

C'est pourquoi, au lieu de dialoguer directement avec un modèle brut, nous allons lui fournir une interface d'exécution très légère (un harnais technique). Depuis votre terminal, vous allez utiliser cette interface pour exploiter l'intelligence du modèle afin d'accomplir des tâches de développement complexes. Plutôt que de soumettre une tâche directement au modèle et de le laisser chercher tant bien que mal des informations éparpillées, nous mettons à sa disposition un ensemble d'outils et un environnement dédié. Ces fonctionnalités permettent au modèle de parcourir méthodiquement les bases de code et de résoudre des problèmes d'une complexité bien supérieure.

Quelles sont précisément ces fonctionnalités ? Elles permettent au modèle de disposer d'une mémoire pour se souvenir des préférences de l'utilisateur, de la structure de la base de code explorée ou de la tâche en cours. Nous offrons également au modèle un environnement au sein duquel il peut identifier les données dont il a besoin, formuler un plan d'action, puis l'exécuter. Avec un minimum de lignes de commande, nous pouvons ainsi capitaliser sur l'intelligence du modèle pour obtenir des résultats remarquables. Avec Claude Code, vous pouvez choisir d'utiliser Opus ou Sonnet, selon la complexité du travail à accomplir, la nature de la tâche et votre abonnement.

Lorsque l'on découvre les capacités de Claude Code, il est tentant de n'y voir qu'un outil destiné à générer de grands volumes de code. Pourtant, nous allons débuter ce cours par l'une de ses fonctionnalités les plus puissantes : sa capacité à **découvrir, expliquer et concevoir**. Avant même de lui demander d'écrire la moindre ligne de code, utilisez Claude Code comme un levier pour monter rapidement en compétences sur une base de code existante. Nous aborderons bien sûr la génération de code, mais nous verrons également comment l'utiliser en dehors du terminal, dans des environnements comme GitHub. Nous détaillerons le refactoring, le débogage d'erreurs et `les cas d'usage où cet outil brille particulièrement`. De plus, il s'avère extrêmement précieux pour l'analyse de données et tout environnement où l'intelligence du modèle peut générer des visualisations percutantes, des ressources ou des livrables de qualité.

Comme nous l'avons mentionné, nous fournissons au modèle un harnais et un environnement pour collecter du contexte et agir, le tout soutenu par une couche de mémoire dont nous détaillerons la structure dans un instant. Attardons-nous d'abord sur les outils et les fonctionnalités natives que nous mettons à la disposition du modèle.

Pour illustrer l'utilisation des outils (*tool use*), imaginez qu'un utilisateur demande quel code est écrit dans un fichier spécifique. Le modèle de langage ne sait pas, par lui-même, naviguer dans une arborescence ou localiser un fichier. C'est ici que les outils interviennent. Clé en main, Claude Code intègre une liste ciblée d'outils, à commencer par la capacité de lire un fichier. Dès que le modèle identifie l'action requise, il appelle cet outil, accède au contenu du fichier et restitue les données à l'utilisateur. Cette capacité à appeler des outils transforme le modèle d'un simple assistant textuel en un agent logiciel extrêmement sophistiqué.

Voici la liste des outils natifs intégrés dans Claude Code :

- Des outils d'édition et de modification pour différents types de fichiers.

- Des outils de lecture de fichiers.

- Des outils d'action avancés pour identifier des patterns (motifs), effectuer des recherches sur le web, ou même créer et exécuter des sous-agents afin de déléguer des tâches particulièrement complexes.

- Enfin, s'agissant d'un outil en ligne de commande, il intègre un outil indispensable pour exécuter des commandes Bash ou Shell.

C'est cet appel d'outils qui permet à Claude Code de rassembler le contexte et les informations nécessaires pour résoudre des problèmes difficiles. Grâce à cette approche, Claude Code **n'a pas besoin d'indexer l'intégralité de votre base de code**, ce qui élimine d'emblée de potentielles failles de sécurité. De 

plus, Claude Code est hautement extensible : vous pouvez lui ajouter de nouveaux outils en le connectant à des serveurs **MCP (Model Context Protocol)**. Le MCP est un protocole open-source et agnostique qui permet aux systèmes d'IA et aux sources de données de communiquer de manière transparente. Ces serveurs MCP enrichissent les fonctionnalités de Claude Code pour une grande variété de tâches professionnelles.

Prenons le temps de préciser ce que signifie l'absence d'indexation de la base de code. Au lieu de générer une représentation structurée et figée de votre code et de l'analyser en permanence, Claude Code utilise une approche appelée **recherche agentique** (*agentic search*). Au lieu d'envoyer l'intégralité de vos fichiers vers un serveur externe — ce qui pourrait vous faire sortir de votre écosystème sécurisé —, Claude Code déploie un ou plusieurs agents équipés d'outils pour localiser précisément l'information recherchée. Ainsi, votre code n'est pas injecté massivement dans le contexte et `ne quitte pas votre environnement,` ce qui répond aux exigences de sécurité les plus strictes des entreprises.

La mémoire de Claude, c'est-à-dire sa capacité à retenir ce qui s'est produit au cours des conversations précédentes ou lors d'actions antérieures, est gérée par un fichier au format Markdown nommé `claude.md`. Dans ce fichier `claude.md`, vous pouvez consigner vos configurations courantes ou vos guides de style. Ce document est automatiquement chargé dans le contexte de l'IA dès le lancement de l'outil. Par ailleurs, l'historique de vos conversations avec Claude Code est stocké localement sur votre machine. 

Vous pouvez réinitialiser ce fil de discussion à tout moment pour ouvrir une nouvelle fenêtre de contexte vierge. À l'inverse, si vous devez reprendre une session précédente ou réactiver un échange antérieur, vous pouvez le faire très simplement.

Faisons un test dans le terminal intégré de VS Code. Nous disposons d'un dossier nommé `demo`, pour l'instant vide. Ouvrons Claude Code en lançant la commande `claude`. S'agissant de la première exécution dans ce répertoire, l'outil nous demande de confirmer que nous faisons confiance aux fichiers de ce dossier, ce que nous validons. Après quelques conseils de prise en main, entrons une instruction simple : "Génère une visualisation sympa pour moi."

Nous constatons immédiatement que Claude Code commence par structurer une liste d'actions à accomplir (*to-do list*). Dans un projet réel, cette liste pourrait consister à chercher dans une base de code, éditer des fichiers, écrire des tests ou extraire des rapports ; ici, il s'agit de concevoir une visualisation. Selon l'orientation choisie par le modèle, il peut s'agir d'effets de particules, de feux d'artifice ou d'autres éléments visuels. Notre objectif est de mesurer la rapidité avec laquelle Claude Code applique des modifications concrètes.

Comme nous opérons directement au sein de Visual Studio Code et que Claude Code bénéficie d'une intégration native avec cet éditeur, nous voyons visuellement les fichiers se créer et se modifier en temps réel. Nous acceptons les modifications, et pour les étapes suivantes, nous pouvons autoriser Claude Code à appliquer ses changements de manière autonome sans nous demander de permission systématique.

Notre visualisation est maintenant entièrement construite. Demandons à Claude Code de l'ouvrir dans notre navigateur internet. L'outil confirme l'action et lance la commande appropriée. Nous découvrons le résultat à l'écran : une interface interactive fonctionnelle. Nous pouvons y ajouter des particules, activer ou désactiver l'animation et vider le rendu. Nous pouvons enrichir cette application autant que nous le souhaitons, modifier ses fonctionnalités ou lui adjoindre de nouveaux modules. Ce test rapide démontre à quel point il est fluide et immédiat de prendre en main cet outil et d'obtenir un livrable opérationnel.

Dans la prochaine leçon, nous explorerons comment utiliser Claude Code à l'échelle d'une base de code de grande envergure, et nous verrons toute sa puissance pour analyser et expliquer des architectures logicielles denses et complexes.

Lesson 3: Compréhension de la configuration et de la base de code

Voici la transcription et la traduction française fluide, professionnelle et entièrement épurée de ses repères temporels.

Le premier exemple sur lequel nous allons travailler est un chatbot RAG (Génération Augmentée de Récupération) de bout en bout. Déployons **Claude Code** pour explorer la base de code.

Avant de commencer à demander à Claude Code d'écrire de grands volumes de code pour nous, voyons comment nous pouvons utiliser cet outil pour monter rapidement en compétences sur des bases de code de grande envergure. Dans notre cas d'école, nous disposons d'une application conçue pour discuter avec Claude à propos des supports de cours de *Deep Learning AI*.

Si nous testons l'interface en lui demandant le plan du cours sur les applications d'IA à contexte enrichi avec le protocole MCP d'Anthropic, nous obtenons une réponse très détaillée structurant les leçons et leurs descriptions respectives. Voyons maintenant comment appréhender le code sous-jacent qui propulse cette application.

De retour dans VS Code, le terminal est ouvert. Lançons Claude Code en tapant simplement la commande `claude`. Dans cet environnement, nous pouvons dialoguer directement avec notre base de code. **Commençons par une question de haut niveau pour obtenir une vue d'ensemble du projet.**

Claude Code ne va pas scanner bêtement chaque fichier individuel. Il déploie une **recherche agentique** pour cartographier la base de code, identifier les fichiers les plus structurants et restituer une synthèse claire de l'architecture, des composants clés et des fonctionnalités de l'application.

Si nous voulons comprendre comment les documents de cours sont ingérés, nous pouvons lui poser la question directement. Notre application utilise une architecture RAG pour extraire les informations pertinentes. On dit souvent que Claude Code est un fantastique ingénieur à vos côtés, mais **c'est un explicateur encore meilleur.** Avant de lui faire rédiger du code, utilisez-le pour comprendre la dynamique du système.

L'outil nous indique précisément quelles portions de code découpent le texte en morceaux (*chunks*), lesquelles injectent le contexte de la leçon et comment sont stockées les métadonnées. Cette capacité à s'approprier un projet, en particulier lorsque vous ne maîtrisez pas la pile technologique ou le langage sous-jacent, est d'une valeur inestimable. Au lieu de naviguer manuellement dans chaque dossier, nous pouvons exiger des analyses spécifiques et même générer des diagrammes.

**Demandons à Claude de retracer l'intégralité du processus de traitement d'une requête utilisateur, du frontend jusqu'au backend (**de bout en bout). Même si vos connaissances sur une partie de la stack sont limitées, Claude Code pallie ce manque.

Pendant que l'outil travaille, il affiche sa liste de tâches (*to-do list*), ce qui permet de suivre sa réflexion en temps réel. À tout moment, vous pouvez appuyer sur la touche `Échap` pour interrompre le processus et réorienter l'agent. Ici, le plan est parfait : il part du frontend, suit les points de terminaison de l'API (endpoints), analyse le moteur RAG et détaille la génération de la réponse. Il inspecte les fichiers cibles et livre un résumé puissant.

Notez que vous pouvez exécuter Claude Code dans le terminal intégré de VS Code ou ouvrir une instance de terminal dédiée. Le résultat affiché est extrêmement précis. Nous comprenons le comportement du JavaScript côté frontend, la réception de la requête par le backend, le filtrage par le système RAG, la recherche vectorielle au sein de la base de données **ChromaDB**, l'application de l'invite système (system prompt) et enfin le renvoi de la réponse finale à l'utilisateur.

**Puisque les représentations visuelles facilitent grandement l'apprentissage, demandons à Claude de dessiner un diagramme pour illustrer ce flux**. Claude Code peut concevoir du rendu pour des visualisations web (via D3.js) ou générer directement de l'art ASCII de qualité dans le terminal.

Le diagramme ASCII généré par l'outil est très propre : il modélise le point d'entrée frontend, l'appel API vers le backend, l'articulation des invites et des outils, la requête vectorielle dans ChromaDB et la synthèse finale par le grand modèle de langage. Vous pouvez enrichir Claude Code avec des outils tiers pour générer des visuels encore plus poussés, mais cette restitution native permet déjà de comprendre l'architecture en une fraction du temps habituellement nécessaire.

Posons-lui maintenant une question pratique : **"Comment puis-je lancer cette application ?"** Claude identifie immédiatement la documentation de l'API, l'interface web et les variables d'environnement requises. Nous sommes prêts à coder.

Examinons à présent l'une des commandes les plus importantes de Claude Code : la commande `/init`. En tapant une barre oblique (`/`), le terminal affiche la liste des commandes natives. La commande `/init` analyse la base de code pour initialiser automatiquement un fichier de mémoire technique appelé `CLAUDE.md`.

Le fichier `CLAUDE.md` est la pièce maîtresse pour doter Claude Code d'une **mémoire à long terme** sur le projet. Il consigne la manière de gérer l'environnement, les commandes de build, de l'interfaçage des tests, du linter et toutes les directives spécifiques que Claude doit respecter à chaque ouverture de session.

Il existe trois types de fichiers `CLAUDE.md` que vous pouvez exploiter :

1. **`CLAUDE.md` (Au niveau du projet) :** Généré par la commande `/init`, il réside à la racine du projet. Il est partagé avec le reste de l'équipe et intégré au contrôle de version (Git). Vous pouvez également en imbriquer dans des sous-dossiers (ex: `/frontend` ou `/backend`) pour des instructions plus locales.
2. **`CLAUDE.local.md` (Local) :** Destiné à vos configurations personnelles (préférences de terminal ou d'éditeur). Ce fichier est configuré pour être ignoré par Git et n'est pas partagé.
3. **`CLAUDE.md` (Global) :** Placé dans le répertoire `.claude` de votre dossier d'accueil utilisateur (*home directory*), il applique ses directives à l'ensemble des projets traités par Claude Code sur votre machine.

Lançons l'initialisation. Comme nous utilisons Claude Code conjointement avec Visual Studio Code, nous voyons le fichier se créer instantanément dans l'éditeur. À chaque action sur le système de fichiers, l'outil sollicite notre validation. Cette approche de l'humain au cœur de la boucle (*Human-in-the-loop*) garantit une sécurité totale. Vous pouvez accepter manuellement ou choisir l'option d'acceptation automatique pour accélérer le processus.

Le fichier `CLAUDE.md` généré contient un aperçu du projet, les technologies clés, une vue d'ensemble de l'architecture et un diagramme des composants. Claude continue d'analyser le code et enrichit le document de manière autonome.

Grâce à la commande `/ide`, Claude Code se connecte nativement à votre éditeur (ici VS Code). Dès que vous visitez un fichier ou inspectez des lignes de code spécifiques, l'éditeur transmet ce contexte en temps réel à Claude Code, facilitant grandement vos requêtes textuelles sur le code ouvert.

Pour amender la mémoire de Claude sans ouvrir manuellement le fichier, vous pouvez utiliser le raccourci d'enregistrement direct (l'icône de livre ou la commande dédiée). 

Spécifions par exemple : *"Utilise toujours uv pour lancer le serveur, n'utilise pas pip directement."* 

L'outil nous demande où stocker cette règle (projet, local ou utilisateur). Choisissons la mémoire du projet pour que toute l'équipe en bénéficie. Le fichier `CLAUDE.md` s'actualise instantanément, et toute mention relative à la gestion des dépendances intègre désormais le gestionnaire **uv**.

En dehors de `/init`, voici les principales commandes intégrées à connaître :

- **/help :** Affiche une description rapide et un résumé de toutes les commandes disponibles.

- **/clear :** Efface l'intégralité de l'historique de la conversation pour ouvrir une fenêtre de contexte totalement vierge. C'est idéal lorsque l'on change de tâche pour éviter de saturer la mémoire du modèle.

- **/compact :** Compresse la session. Cette commande vide l'historique détaillé des échanges mais conserve un résumé des actions accomplies, permettant de poursuivre le développement sur une base de contexte allégée.
- **La touche `Échap` :** Interrompt immédiatement n'importe quel processus ou génération en cours pour vous redonner la main sur le terminal.

Pour conclure cette étape, voyons comment Claude Code s'interface nativement avec **Git**. Nous avons modifié notre configuration de projet et nous devons indexer et valider ces changements. Au lieu de taper manuellement les commandes Git et de rédiger péniblement un message de validation, demandons à Claude Code de s'en charger.

L'agent analyse les différences de code (*diffs*), indexe les fichiers et génère un message de validation (*commit message*) particulièrement descriptif et normé. C'est un atout précieux pour l'historique du projet et pour vos futurs collaborateurs sur GitHub. Notre commit est prêt.

Dans la prochaine leçon, nous allons passer à la vitesse supérieure et utiliser Claude Code pour écrire de grands volumes de fonctionnalités de manière totalement automatisée.

Lesson 4: RAG - Notre chatbot RAG de bout en bout

Vue globale du projet

RAG Chatbot pour cours en ligne — un assistant qui répond à des questions sur des scripts de cours, en cherchant dans une base vectorielle plutôt qu'en s'appuyant uniquement sur la mémoire de Claude.

---

Ce que c'est

Un système full-stack minimaliste en deux parties :

- Backend Python (FastAPI) — indexe des fichiers texte de cours, expose une API de question/réponse
- Frontend vanilla JS — interface de chat avec sidebar de stats et rendu markdown

---

La stack technique

┌──────────────┬────────────────────────────────────────┐
│    Couche    │              Technologie               │
├──────────────┼────────────────────────────────────────┤
│ Langage      │ Python 3.13+, JavaScript               │
├──────────────┼────────────────────────────────────────┤
│ Web          │ FastAPI + Uvicorn                      │
│ framework    │                                        │
├──────────────┼────────────────────────────────────────┤
│ Vector DB    │ ChromaDB (persistance disque)          │
├──────────────┼────────────────────────────────────────┤
│ Embeddings   │ all-MiniLM-L6-v2                       │
│              │ (sentence-transformers, local)         │
├──────────────┼────────────────────────────────────────┤
│ LLM          │ Claude via Anthropic SDK               │
│              │ (claude-sonnet-4-20250514)             │
├──────────────┼────────────────────────────────────────┤
│ Package      │ uv                                     │
│ manager      │                                        │
└──────────────┴────────────────────────────────────────┘

---

Comment ça fonctionne

1. Au démarrage, le serveur parse les fichiers docs/*.txt et les découpe en chunks de 800 caractères avec 100 de chevauchement, puis les encode en vecteurs et les stocke dans ChromaDB
2. Quand l'utilisateur pose une question, le backend l'envoie à Claude avec un outil search_course_content disponible
3. Claude décide s'il doit chercher — si oui, il appelle l'outil, qui fait une recherche sémantique dans ChromaDB et retourne les 5 chunks les plus proches
4. Claude génère la réponse finale à partir de ces chunks
5. Le frontend affiche la réponse en markdown avec les sources repliables

## 2. Développement Parallèle Isolé : Le Pattern Git Worktree

Lorsque vous devez mener de front le développement de plusieurs fonctionnalités distinctes (par exemple : ajuster l'interface graphique, étendre la couverture de tests API et configurer l'infrastructure DevOps), ouvrir de simples terminaux en parallèle dans le même répertoire s'avère extrêmement risqué. Si deux instances de l'IA manipulent simultanément ou de façon croisée les mêmes fichiers, vous ferez face à des écrasements de code, des corruptions d'état et des régressions complexes.

Pour isoler totalement le raisonnement et les actions de vos agents, la bonne pratique consiste à coupler Claude Code avec les **arbres de travail Git** (`Git worktrees`). Un *worktree* vous permet de lier plusieurs répertoires de travail indépendants à un unique dépôt Git local. Chaque dossier possède sa propre branche active et ses fichiers physiques isolés, éliminant tout risque d'interférence.

Déployons trois chantiers en parallèle en créant un dossier masqué `.trees` à la racine de notre projet, puis initialisons nos arbres de travail :

Bash

```
git worktree add .trees/ui_feature -b feature/ui-theme
git worktree add .trees/test_feature -b feature/api-tests
git worktree add .trees/quality_feature -b feature/code-linting
```

Nous divisons notre terminal en trois fenêtres distinctes, chacune positionnée dans l'un de ces sous-répertoires, puis nous lançons une instance de `claude` dans chaque environnement. Grâce à cette isolation physique, nos agents peuvent travailler en simultané sur des tâches orthogonales ou manipuler les mêmes fichiers de configuration sans jamais s'entrechoquer :

1. **Terminal 1 (UI) :** Nous appelons notre commande personnalisée `/implement-feature` pour concevoir un bouton de bascule entre un thème sombre et un thème clair en JavaScript et CSS.

1. **Terminal 2 (Tests) :** Nous ordonnons à l'agent d'étendre la couverture de tests unitaires sur nos points de terminaison FastAPI.

1. **Terminal 3 (Qualité) :** Nous demandons l'intégration d'outils de qualité de code, impliquant l'écriture de configurations spécifiques pour le linter et le formateur **Black**.

## 4. Cycle de Vie Programmatique : Implémentation des Hooks

L'une des fonctionnalités avancées de Claude Code est la capacité d'intercepter et de conditionner le cycle de vie de l'agent via des **Hooks** (crochets d'exécution). Semblables aux hooks Git, ils vous permettent d'injecter des scripts ou des commandes shell arbitraires avant ou après l'activation d'un outil par l'IA.

Pour configurer graphiquement ou inspecter ces événements, invoquez la commande :

Bash

# 

```
/hooks
```

> ⚠️ **Avertissement de Sécurité Crucial :** L'activation de commandes shell arbitraires via les hooks requiert une vigilance absolue. Vous devez valider scrupuleusement les scripts exécutés pour éviter tout effet de bord ou faille de sécurité au sein de votre environnement de développement.
> 

### Événements Hooks Claude Code : BeforeToolUse et PostToolUse

L'architecture des hooks Claude Code permet de se brancher sur de multiples points de synchronisation du cycle de vie de l'agent :

- `BeforeToolUse` : S'exécute avant l'appel d'un outil (permet d'interdire ou de conditionner l'usage d'une commande selon le contexte).
- `PostToolUse` : S'exécute immédiatement après l'utilisation réussie d'un outil.
- Événements liés aux notifications, aux soumissions d'invites ou avant la conclusion de la réponse d'un sous-agent.

### Implémentation d'un Hook `PostToolUse`

Pour matérialiser ce concept, nous configurons un hook qui s'activera exclusivement lorsque Claude Code appelle les outils natifs de lecture (`view_file`) ou de recherche (`grep`). Nous stipulons que dès que l'action se termine, le terminal doit exécuter la commande système vocale `say "Tout est fait !"`.

Cette configuration est automatiquement sérialisée et stockée localement dans votre fichier de configuration de projet `.claude/settings.local.json` :

JSON

# 

```
{
  "hooks": {
    "PostToolUse": {
      "matcher": "view_file|grep",
      "cmd": "say 'Tout est fait !'"
    }
  }
}
```

Testons l'implémentation. Nous relançons Claude Code et lui demandons de lire le script de démarrage `run.sh`. L'agent appelle l'outil de lecture, affiche le contenu à l'écran, et instantanément, notre machine émet l'alerte vocale *"Tout est fait !"*, confirmant le déclenchement du hook.

Au-delà de cet exemple ludique, les hooks de grade production permettent d'automatiser des tâches MLOps et DevOps indispensables : exécuter un linter ou un formatage automatique dès qu'un fichier est modifié, lancer une suite de tests pytest après chaque réécriture de fonction, ou forcer une phase d'auto-revue par l'IA avant de valider une modification d'infrastructure.

## 1. Diagnostic d'un Notebook Déstructuré

Dans le domaine de la *Data Science*, les notebooks Jupyter sont d'excellents outils d'exploration, mais ils tendent à devenir désorganisés, denses et difficiles à maintenir en production.

Bien que le notebook affiche les résultats attendus, l'architecture souffre de plusieurs défauts majeurs :

Pour industrialiser ce script, notre objectif est d'extraire la logique métier vers des modules Python orientés objet et de générer un notebook épuré, configuré et hautement visuel.

## 2. Refactoring et Séparation des Préoccupations (*Separation of Concerns*)

4. La recherche agentique permet à Claude de lire les fichiers au besoin au lieu d'indexer l'ensemble de la base de code, ce qui réduit les risques de sécurité liés à l'exposition des données.

    Correct

    La recherche d'agents évite d'envoyer l'ensemble de la base de code ailleurs en ne lisant que les fichiers pertinents en cas de besoin, ce qui permet de minimiser le risque de fuite de données sensibles.

### **Caractéristiques supplémentaires de Claude**

- **Mode de réflexion étendu**

    Pour les tâches complexes (par exemple, les changements architecturaux complexes, le débogage de problèmes compliqués), vous pouvez utiliser le mot "penser" pour déclencher **le mode de réflexion étendue**. Il y a plusieurs niveaux de pensée : "penser" < "penser dur" < "penser plus fort" < "ultrapenser". Chaque niveau alloue plus de budget de réflexion pour Claude.

- **Utilisation de sous-agents**

    Vous avez appris que l'un des outils prés à l'emploi pour Claude Code est **Task**, que Claude Code peut utiliser pour lancer des sous-agents pour des tâches complexes en plusieurs étapes. Vous pouvez explicitement demander à Claude Code d'utiliser des sous-agents pour réfléchir à des idées ou pour enquêter sur de multiples aspects d'une question ou d'un problème que vous souhaitez résoudre. Ces agents intégrés sont à usage général.

    Vous pouvez également créer vos sous-agents spécialisés personnalisés. Chaque sous-agent a sa propre fenêtre de contexte, et vous pouvez définir une invite système personnalisée et des outils spécifiques pour chaque sous-agent. Cette partie n'est pas couverte par ce cours, mais vous pouvez vérifier les détails dans la documentation **ici**.
