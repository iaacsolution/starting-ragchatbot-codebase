Lesson 1: Pourquoi les Agent Skills - Démonstration et cas d'usage

La leçon part d'un cas concret : analyser des données de campagne marketing CSV et générer un rapport Excel structuré avec recommandations de réallocation budgétaire. La skill analyzing-marketing-campaign produit un rapport Excel avec analyse, graphiques et recommandations — généré automatiquement à chaque nouvelle semaine de données. Depuis le 26 janvier 2025, les skills sont disponibles sur Claude.ai plan gratuit (Sonnet 4.5 ou Haiku 4.5). Les skills sont un standard ouvert utilisables sur toute app compatible (agentskills.io). Pour créer ses propres skills, il faut se demander : y a-t-il des instructions que tu demandes en boucle à un agent ? Peux-tu les rassembler dans une skill unique ?

Lesson 2: Définition complète des Agent Skills

Une skill est un dossier de fichiers organisé contenant : instructions (SKILL.md), scripts exécutables, assets (images, templates, icônes), et références (guides de marque, règles métier). L'agent peut charger ces éléments à la demande pour accomplir des tâches spécifiques avec précision. Les skills sont chargées depuis le système de fichiers local. Avant les skills, on avait des agents spécialisés (un par domaine) avec un échafaudage complexe. Maintenant : un agent général + skills chargées à la demande avec un échafaudage simple (bash + système de fichiers). Les 3 catégories de ce que les skills apportent : expertise du domaine (directives de marque, processus juridique, méthodologies d'analyse), flux de travail répétables (bilan hebdomadaire campagne marketing, préparation appels clients), nouvelles capacités (créer des présentations, générer Excel/PDF, construire des serveurs MCP). Structure d'une skill : SKILL.md est le seul fichier requis. Les skills peuvent inclure des scripts exécutables, pas juste du texte. Propriétés clés : portable (la même skill fonctionne sur Claude Code, Claude.ai, Claude Agent SDK, API Claude, Codex, Gemini CLI — Agent Skills est un standard ouvert comme MCP, pas propriétaire Anthropic) et composable (combiner plusieurs skills pour des workflows complexes). Divulgation progressive : les skills peuvent être des centaines dans un système. Pour ne pas saturer la fenêtre de contexte, 3 couches : métadonnées (nom + description YAML) toujours chargées au démarrage, instructions (contenu SKILL.md) quand la skill est déclenchée, ressources (fichiers, scripts) chargées au besoin uniquement. Les scripts sont exécutés indépendamment de la fenêtre de contexte et ne polluent pas les tokens.

Lesson 3: Skills vs MCP, Outils et Sous-agents

Skills vs MCP : MCP connecte l'agent à des systèmes et données externes, les Skills apprennent à l'agent quoi faire avec ces données. MCP fournit l'accès, les Skills fournissent l'expertise. Analogie : MCP = apporter tous les ingrédients et ustensiles, Skills = la recette pour assembler ces ingrédients de manière reproductible. Skills vs Outils : les outils sont des capacités essentielles pour accomplir des tâches (toujours en live dans la fenêtre contextuelle), les skills sont des connaissances spécialisées et scripts à la demande (chargées dynamiquement). Si un outil n'est pas nécessaire à chaque conversation, l'embarquer dans une skill + divulgation progressive évite de surcharger le contexte inutilement. Skills vs Sous-agents : les sous-agents ont un contexte isolé + autorisations propres et travaillent en parallèle en retournant les résultats au parent. Les skills indiquent comment faire le travail. Les Skills peuvent améliorer à la fois l'agent principal ET ses sous-agents. Exemple customer insights analyzer : MCP Google Drive fournit l'accès aux données, 2 sous-agents analysent en parallèle (entretiens + enquêtes), les skills guident comment catégoriser et résumer. La pyramide de l'écosystème IA : Prompts (unité de base) → Skills (regroupent prompts + code + ressources de façon reproductible) → Sous-agents (délèguent des tâches, utilisent des skills) → Outils / MCP (capacités et données). La fenêtre de contexte est un bien public précieux : minimiser la surcharge avec des sous-agents, charger les données MCP à la demande, et utiliser le chargement progressif des skills uniquement quand nécessaire.

Lesson 4: Skills pré-construites et Skill Creator

Le dépôt officiel github.com/anthropic/skills contient deux catégories : Document skills (Word, PDF, PowerPoint, Excel — toujours actives dans Claude AI) et Example skills (Marketing, Brand, Skill Creator — désactivées par défaut sauf skill-creator). La skill PowerPoint contient SKILL.md avec vue d'ensemble, ce que l'utilisateur peut demander, scripts à exécuter si nécessaire (chargés à la demande, pas au démarrage), principes de design, exigences, palettes de couleurs par défaut. Le Skill Creator est une méta-skill qui crée d'autres skills programmatiquement. Processus en 3 étapes : fournir des exemples concrets du résultat attendu, planification (Claude fait du pattern matching), initialisation (exécution des scripts Python sous-jacents). Les 3 scripts : init_skill.py (crée les fichiers texte à partir d'un template YAML), package_skill.py (zippe la skill avec la bonne arborescence), validate_skill.py (vérifie SKILL.md, frontmatter YAML, structure dossiers). Workflow complet MCP + Skills custom + Skills intégrées pour générer un rapport marketing PowerPoint depuis BigQuery : 3 composants assemblés (skill marketing modifiée pour BigQuery, skill brand guidelines créée, skill PowerPoint intégrée). Bonne pratique : dans le SKILL.md, toujours spécifier le serveur MCP et le nom de l'outil quand une skill en dépend. Note : MCP local (ex: BigQuery) = Claude Desktop uniquement. Claude.ai ne supporte que les serveurs MCP distants.

Lesson 5: Création de skills personnalisées et bonnes pratiques

Structure d'un SKILL.md — deux parties : YAML Frontmatter (métadonnées en haut) et Corps Markdown (instructions). Champs obligatoires : name (max 64 chars, minuscules, chiffres, tirets, recommandé verbe-ing, = nom du dossier parent) et description (max 1024 chars, expliquer ce que fait la skill ET quand l'utiliser, inclure les mots-clés déclencheurs). Champs facultatifs : license, compatibility (max 500 chars), metadata (paires clé-valeur libres), allowed_tools (liste d'outils pré-approuvés). Corps du SKILL.md : moins de 500 lignes (externaliser le reste dans des fichiers référencés), instructions étape par étape, spécifier les cas limites, utiliser des barres obliques pour les chemins (même sous Windows), terminologie cohérente. Degrés de liberté : haute liberté (instructions générales, résultats créatifs), liberté moyenne (pseudocode ou templates), faible liberté (scripts spécifiques, séquence exacte obligatoire, résultats prévisibles). Pour des workflows reproductibles → faible liberté. Répertoires optionnels : assets/ (templates, images, logos), references/ (documentation supplémentaire, lue si besoin — si plus de 100 lignes ajouter une table des matières), scripts/ (code exécutable — préciser si Claude doit exécuter le script ou le lire comme référence). Valider ses skills avec le Skill Creator dans Claude Code : installer avec /plugin install skill-creator, vérifier avec /skills. Évaluation en parallèle avec sous-agents : Claude Code lance 2 sous-agents simultanément, chacun analysant une skill. Tests unitaires pour les skills : format JSON avec skills, queries, files, expected_behavior. Bonnes pratiques : recueillir des feedbacks humains, tester sur tous les modèles prévus, pour les skills avec scripts tester seulement l'orchestration.

Lesson 6: Utiliser les skills dans Claude Code avec sous-agents spécialisés

Les skills de projet vivent dans .claude/skills/ — chaque skill dans son propre dossier avec un SKILL.md. Les skills globales se placent dans le home directory. Depuis janvier 2025, slash commands et skills ont été fusionnés : une skill à .claude/skills/review/SKILL.md crée automatiquement la commande /review. Les Skills offrent en plus : un dossier de ressources et des options frontmatter avancées. Lister les skills disponibles : /skills (liste les skills avec leur nom, description et coût en tokens). Si une skill n'est pas détectée : redémarrer Claude Code après ajout/modification d'une skill. Invocation : model-invocée (Claude décide quand déclencher la skill) ou user-invocée (taper /adding-cli-command manuellement). Frontmatter pour contrôler : disable-model-invocation, user-invocable, allowed-tools, model, argument-hint, context, agent. Sous-agents avec skills : chaque sous-agent opère dans sa propre fenêtre de contexte isolée, transmet son rapport à l'agent principal, puis disparaît. Important : les sous-agents n'héritent pas des skills du parent — il faut les lister explicitement dans leur configuration. Créer un sous-agent via /agents : nom, description/prompt, modèle, outils, couleur, skills. Contrairement à l'agent principal (divulgation progressive), le sous-agent reçoit l'intégralité du SKILL.md préchargée dans son contexte au démarrage. Exécuter une skill dans un sous-agent avec context: fork pour isoler dans un sous-agent. Sous-agents built-in de Claude Code : Explore (recherche et navigation dans le code), Plan (planification architecturale), General-Purpose (tâches générales, défaut pour context: fork).

Lesson 7: Utiliser les skills via l'API Claude

Les 2 règles de base : les skills créées dans Claude AI/Desktop ne sont pas partagées automatiquement avec l'API Claude ou Claude Code (il faut les uploader manuellement via la Files API). Pour qu'une skill fonctionne, Claude doit pouvoir exécuter du code. Dans Claude AI/Desktop, ces autorisations sont activées par défaut. Dans l'API Messages, il faut les configurer manuellement. Différences sandbox API vs Desktop : Claude AI/Desktop a accès aux fichiers par défaut, peut installer des paquets et accéder à internet. L'API Messages (sandbox) doit être configurée pour l'accès fichiers, n'installe pas de paquets (bibliothèques préinstallées), et n'a pas accès internet. Code Execution Tool donne à Claude accès à un terminal Bash pour exécuter des commandes et un système de fichiers pour créer, lire, modifier des fichiers. Files API permet de charger, stocker et récupérer des fichiers dans le conteneur d'exécution. Workflow : uploader le fichier source pour obtenir un file_id, Claude analyse et génère la sortie dans le conteneur, télécharger le fichier de sortie via son file_id. Configurer l'appel API avec les bêta headers : betas=["skills-2025-01", "files-api-2025-01", "code-execution-2025-01"], container={"skill_id": skill_id}, tools=[{"type": "code_execution"}]. Lister les skills : client.skills.list(source="custom"). Analyser la réponse : type text pour le texte, server_tool_use quand Claude exécute du code, bash_code_execution_tool_result pour le résultat contenant le file_id de sortie. Ce que fait Claude sous le capot (divulgation progressive) : lit SKILL.md, charge uniquement les ressources nécessaires, exécute les scripts désignés, génère le fichier de sortie dans le conteneur, renvoie le file_id. Combiner skill personnalisée + skill native Anthropic (docx) : spécifier container={"skills": [custom_skill_id, "docx"]}. Coût estimé : environ 0.67$ pour exécuter toutes les cellules du notebook une fois.

Lesson 8: Utiliser les skills avec le Claude Agent SDK

Architecture : un agent superviseur coordonne 3 sous-agents spécialisés dans leur propre fenêtre de contexte isolée. Main Agent (orchestrateur) avec TaskTool, SkillTool, MCP Notion. Docs Researcher avec WebSearch, WebFetch. Repo Analyzer avec Bash, WriteFile. Web Researcher avec WebSearch, WebFetch. La skill learning-a-tool guide l'orchestrateur principal sur la méthodologie de recherche et le format de restitution. Phase 1 (Investigation parallèle) : déploiement simultané des 3 sous-agents. Phase 2 (Structuration progressive) : synthèse en parcours d'apprentissage modulaire avec aperçu et motivation, installation, concepts fondamentaux, patterns pratiques, cas d'usage industriels. Installation : uv add claude-agent-sdk python-dotenv asyncio. L'agent principal utilise ClaudeAgent avec system_prompt, allowed_tools (incluant TaskTool pour déléguer aux sous-agents et SkillTool pour indexer les SKILL.md), agents (registre des sous-agents), mcp_servers, setting_sources (localise .claude/skills). Chaque sous-agent défini avec AgentDefinition contenant description, prompt, tools. L'orchestrateur détecte la skill et affiche le plan, puis déploie les 3 sous-agents en parallèle et synthétise via divulgation progressive. Syndication vers Notion via MCP : l'agent lit le Markdown local, convertit chaque élément en Notion Rich Blocks natifs et pousse par lots. Sécurité et human-in-the-loop : pour un déploiement production, implémenter un mécanisme d'interruption interceptant tout appel à un outil d'écriture ou de terminal, demander une confirmation humaine explicite. Coût estimé : environ 3.43$ avec Haiku pour les sous-agents, environ 6.35$ avec Sonnet. Durée : environ 15 minutes.
