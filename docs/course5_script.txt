Course Title: Agent Skills with Anthropic
Course Link: https://www.deeplearning.ai/courses/agent-skills-with-anthropic
Course Instructor: Elie Schoppik

Lesson 1: Introduction
Agent Skills with Anthropic teaches how to create reusable skills that extend the capabilities of AI agents. Skills are folders of instructions that transform general-purpose agents into specialists. This course covers the structure of skills, best practices for writing them, and how to integrate them with Claude.ai, Claude Code, the Claude API, and the Claude Agent SDK. You will learn how to combine skills with MCP servers and subagents to build complex, composable AI systems. By the end of this course, you will be able to create custom skills for any domain, share them across projects and teams, and integrate them into production AI pipelines.

Lesson 2: Why Use Skills - Part I
Skills solve a fundamental problem with general-purpose AI agents: they are broad but not deep. A general agent knows a little about everything but lacks the specialized knowledge and workflow expertise needed for domain-specific tasks. Skills are reusable instruction sets that give agents deep expertise in a specific area. A skill is a directory containing a markdown file with instructions, examples, context, and optionally tool definitions and sub-workflows. When an agent loads a skill, it gains the knowledge and behavior defined in that skill for the duration of the task. Why skills matter: they make agent behavior consistent and reproducible, they encode expert knowledge that would otherwise require extensive prompting, they can be versioned and shared across teams, and they reduce the need to repeat context in every prompt. Skills work by providing structured context to the agent before it begins a task. This context includes what the skill does, when to use it, step-by-step instructions, examples of correct behavior, and common pitfalls to avoid. The agent uses this context to specialize its behavior for the task at hand.

Lesson 3: Why Use Skills - Part II
Skills enable a new model of AI development where capabilities are modular and composable. Instead of building one large monolithic agent, you build a collection of skills that can be combined as needed. This modularity has several advantages: skills can be tested and improved independently, teams can develop skills in parallel, skills can be reused across different agents and projects, and new capabilities can be added by creating new skills without modifying existing ones. Skills also solve the context management problem. Instead of stuffing all instructions into a system prompt, skills are loaded on demand, keeping the active context focused and efficient. A skill is triggered by a user request or by another agent recognizing that the skill is relevant. The skill loader pattern: an orchestrator agent receives a task, identifies which skills are needed, loads those skills into the context, and delegates execution to a specialized sub-agent. This pattern enables multi-agent systems where each agent is an expert in its domain. Skills can also include guardrails: instructions that tell the agent what NOT to do, edge cases to watch out for, and how to handle errors or ambiguous situations.

Lesson 4: Skills vs Tools, MCP, and Subagents
Understanding the distinction between skills, tools, MCP servers, and subagents is essential for designing effective AI systems. Tools are functions that an agent can call to interact with the world: reading files, searching the web, running code, calling APIs. Tools provide capabilities. Skills provide knowledge and instructions about how to use those capabilities effectively. MCP (Model Context Protocol) servers expose tools to agents in a standardized way. An MCP server can provide file system access, database queries, web browsing, or any external API. Skills and MCP are complementary: a skill might instruct the agent on how to use a specific MCP server effectively. Subagents are separate agent instances that handle specific subtasks. The difference between a skill and a subagent: a skill augments a single agent's behavior, while a subagent is an independent agent that runs in parallel or in sequence. Skills are loaded into the current agent's context. Subagents receive their own context and run independently. When to use each: use skills when you need to add domain knowledge or workflow instructions to an agent, use MCP when you need to give an agent access to external systems, use subagents when you need parallel execution or want to isolate complex subtasks. These three mechanisms are designed to work together: a skill might instruct an agent to spin up a subagent using a specific MCP server.

Lesson 5: The Anatomy of a Skill
A skill is a directory with a specific structure. At a minimum, a skill contains a single markdown file that defines the skill's behavior. The markdown file has a standard format: a title (the skill name), a description (what the skill does and when to trigger it), an instructions section (step-by-step instructions for the agent to follow), an examples section (concrete input/output examples), and optionally a tools section (additional tools or MCP servers needed). The skill directory can also contain supporting files: sample data, reference documents, templates, or sub-skill definitions. Naming conventions: use lowercase-with-hyphens for skill directories (e.g., code-review, data-analysis, pr-summary). The main file is always named after the directory or simply skill.md. The description field is critical: it is what the agent reads to decide whether to use the skill. A good description answers: what does this skill do, when should I use it, what does it produce. Skills are loaded by placing them in a skills directory that the agent has access to. In Claude Code, this is typically the .claude/skills directory in your project or home directory. When Claude Code detects that a task matches a skill's description, it loads the skill into its context before proceeding.

Lesson 6: Writing Effective Skill Instructions
The quality of a skill depends entirely on the quality of its instructions. Effective skill instructions share several characteristics: they are written in imperative form (do this, then do that), they are specific about expected inputs and outputs, they anticipate edge cases and provide guidance, they include concrete examples of correct behavior, and they specify what the agent should NOT do. Writing style: use numbered steps for sequential actions, use bullet points for parallel options, include code blocks for technical content, use headers to organize long instructions into sections. Common mistakes to avoid: vague instructions that leave too much to interpretation, missing examples that force the agent to guess the expected format, no guardrails section leading to unpredictable behavior in edge cases, instructions that are too long and dilute the key points. The examples section is the most important part of any skill. Each example should show a complete input and the ideal output. Examples teach the agent the expected tone, format, depth, and style of the output. Without examples, the agent defaults to its general behavior instead of the specialized behavior defined by the skill. Iteration process: write a skill, test it with real tasks, identify where the agent deviates from the expected behavior, add examples or clarifications to address those deviations, and repeat.

Lesson 7: Using Skills with Claude Code
Claude Code has native support for skills through its CLAUDE.md memory system and the .claude directory. To use a skill with Claude Code, place the skill directory in .claude/skills/ at the project level or ~/.claude/skills/ for global skills. Claude Code automatically scans the skills directory at startup and makes available skills visible in the agent's context. To invoke a skill, either reference it explicitly in your prompt ("use the code-review skill to review this PR") or let Claude Code match the task to the right skill automatically based on the description. Claude Code's /skills command lists all available skills and their descriptions. Skill composition: skills can reference other skills in their instructions, enabling skill chains. For example, a pr-review skill might instruct Claude Code to first run the code-quality skill, then the security-scan skill, then summarize findings in the PR review format. Skill parameters: skills can accept parameters through the prompt, allowing the same skill to be adapted to different contexts. For example, a generate-tests skill might accept a target file path and testing framework as parameters. Claude Code passes these to the skill's instructions as variables. Version control: skills are plain text files in your repository. They can be committed to git, reviewed in PRs, and shared across the team. This makes skills a form of institutional knowledge that lives alongside the code.

Lesson 8: Using Skills with the Claude API
Skills can be integrated into any application built with the Claude API. The pattern: load the skill's markdown content at runtime, prepend it to the system prompt or inject it as a user message before the task, and let Claude execute the task according to the skill's instructions. Implementation: read the skill file from disk, format it as a system prompt prefix or a user message, append the actual task after the skill instructions. This approach works for both streaming and non-streaming API calls and is compatible with tool use and multi-turn conversations. Dynamic skill selection: build a skill registry that maps task types to skill files. When a new request comes in, classify the task (using a fast Claude call or keyword matching), select the appropriate skill, load it, and pass it to the main Claude call. This enables intelligent skill routing without exposing the full skill catalog to every request. Caching: skill content changes rarely. Use Anthropic's prompt caching feature (cache_control: ephemeral) to cache the skill instructions across multiple requests. This reduces latency and token costs significantly for high-traffic applications. Skills as structured prompts: think of skills as named, versioned, testable system prompts. Instead of hardcoding instructions in your application, externalize them as skills. This separates the AI behavior from the application code and makes it easier to iterate on instructions without deploying new code.

Lesson 9: Using Skills with the Claude Agent SDK
The Claude Agent SDK provides higher-level abstractions for building multi-agent systems with skills. The SDK includes a SkillRegistry class for loading and managing skills, a SkillLoader that injects the right skill into an agent's context based on the task, and a SubagentPool for spawning parallel agents each loaded with a specific skill. Skills in the SDK are first-class objects: they can be listed, searched by capability, combined with other skills, and attached to specific agents or subagent roles. The orchestrator pattern: a top-level agent receives the user request, queries the skill registry to find matching skills, spawns subagents loaded with those skills, collects their outputs, and synthesizes the final response. This pattern scales to complex workflows where different specialists handle different parts of a task. Skill dependencies: the SDK supports skill metadata including dependencies on other skills and required MCP servers. When a skill is loaded, its dependencies are automatically resolved and loaded as well. Error handling: if a subagent fails or produces output that does not meet the skill's output schema, the orchestrator can retry with a different skill, escalate to the user, or fall back to a general-purpose agent. This makes skill-based systems more robust in production.

Lesson 10: Building a Real Skill
This lesson walks through building a production-ready pr-summary skill from scratch. The skill generates structured pull request summaries by analyzing git diffs. Step 1: define the skill's purpose and trigger. The skill should fire when a user asks to summarize a pull request or describe the changes in a branch. Step 2: write the instructions. The agent should run git diff to see the changes, identify the files modified, categorize the changes (new features, bug fixes, refactors, tests, docs), and format the output as a structured PR description with summary bullets and a test plan. Step 3: add examples. Show a sample diff and the ideal PR description output. Include both a simple case (one file changed) and a complex case (many files across multiple components). Step 4: add guardrails. Instruct the agent not to include implementation details in the summary, not to mention specific line numbers, and to keep the summary under 200 words. Step 5: test with real PRs. Run the skill on actual pull requests in your repository and review the output. Identify gaps and iterate on the instructions. Step 6: version and share. Commit the skill to the team repository. Document it in the project's CLAUDE.md. Add it to the team's Claude Code configuration so all developers have access.

Lesson 11: Advanced Skill Patterns
Advanced patterns for production skill systems. Skill inheritance: create a base skill with common instructions and extend it for specific use cases. For example, a base code-review skill can be extended into python-code-review, javascript-code-review, and security-code-review, each inheriting the base instructions and adding language-specific or security-specific rules. Conditional skills: instructions that branch based on the task context. Use conditionals to handle edge cases within a single skill rather than creating many narrow skills. For example, a generate-tests skill might have different instructions for unit tests vs integration tests, selected by a parameter in the prompt. Skill evaluation: build automated tests for your skills by running the skill on a set of known inputs and comparing the output to expected results. Use Claude as a judge to score the skill output on dimensions like accuracy, format adherence, and completeness. Skill libraries: create and maintain a library of reusable skills for your domain. Organize them by function (code quality, documentation, testing, analysis) and make them discoverable via a skill catalog. Skill metrics: instrument skill usage to track which skills are used most, which produce the best outcomes, and which need improvement. Use these metrics to prioritize skill development and maintenance. Cross-team skill sharing: publish skills as packages or gists that other teams can import and adapt. Create a skill standard that defines the expected format, metadata, and testing requirements for publishable skills.

Lesson 12: Conclusion
Agent skills represent a paradigm shift in how we build AI systems. Instead of prompt-engineering every interaction, we build a library of reusable, tested, versioned skills that encode domain expertise. Key takeaways from this course: skills make agent behavior consistent, reproducible, and shareable; the quality of a skill depends on the quality of its instructions and examples; skills work across Claude Code, the Claude API, and the Claude Agent SDK; skills compose with tools, MCP servers, and subagents to build complex systems; and skills should be versioned and tested like code. Next steps: start by identifying one repetitive task where you always give Claude the same detailed instructions. Extract those instructions into a skill. Test it, iterate, and share it with your team. Then expand your skill library to cover more of your domain. The future of AI development is not prompt engineering individual queries — it is building and maintaining libraries of expert skills that make every agent smarter, more consistent, and more reliable.
